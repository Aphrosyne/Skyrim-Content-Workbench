# 一、项目背景

## 项目目标（最初设想）

希望能做到一个多功能集合的本地内容管理器，Skyrim 本地静态 Mod 特化。

既能展开现有的文件树形分类结构，让手动整理变得便捷；又能利用预览图帮助识别 Mod 内容；还能通过 AI JSON 交换辅助批量分类。

## 开发背景

一开始使用 Windows 资源管理器加上已做好的文件分类结构进行手动分类整理。

但 Win 资源管理器存在以下问题：

- 文件夹层级太深时，在不同文件夹之间跳来跳去需要大量鼠标点击
- 树形交互不够直观，需要反复返回上层再进入目标目录
- 预览图模式不好用，对服装/护甲类 Mod 辨认帮助有限
- 不是专为 Mod 目录设计的，无法提供特化的交互体验

# 二、最初希望解决的问题

以下内容全部来源于项目初期需求讨论（见 `docs/reference-conversation.md`），按问题维度整理，**不代表当前设计决策**。

---

## 问题一：Windows 资源管理器的文件夹跳转效率太低

### 问题描述

Win 资源管理器在多层级文件夹之间的跳转非常繁琐。用户已有按类型分类的树形目录结构，需要在不同分类目录之间切换来移动 Mod 文件。

### 实际痛点

- 文件夹太多层时，每次在不同文件夹之间交换位置就特别累
- 总得返回上层目录，然后重新双击多次才能进入目标位置
- 用户的文件结构是树状的，但资源管理器的操作方式不适合树状跳转

### 期望的解决方向

一个可交互的树状目录视图，能够减少文件夹跳转的点击次数，在树状图中快速定位目标目录。

---

## 问题二：缺少有效的预览图帮助识别服装/护甲 Mod

### 问题描述

Skyrim 大量 Armors & Clothes Mod 仅靠分类无法辨认，需要看到预览图才知道 Mod 长什么样。Win 资源管理器的预览图模式不够好用。

### 实际痛点

- 大量 mod 仅凭英文文件名无法判断内容
- 时间长记不住英文 mod 名字对应什么内容，需要去 N 网站查尾号才能回想起来
- 部分 mod 来自社区分享，文件名没有标准格式（不包含 N 网尾号），更难追溯
- N 网下载的 Webp 预览图文件名是一串数字码，无法通过文件名关联

### 期望的解决方向

Mod 列表必须关联预览图。理想情况下能自动从 N 网获取预览图，退而求其次可通过手动按钮导入关联。

---

## 问题三：手动分类整理 Mod 的点击操作太繁琐

### 问题描述

用户有大量未分类的 Mod 文件，需要人工整理到已有的分类目录结构中。在资源管理器中完成这一流程涉及大量重复的"返回 → 进入 → 返回 → 再进入"操作。

### 实际痛点

- 在不同文件夹之间搬动文件的操作特别累、特别费时间
- 用户已经有一个按 Mod 类型分类的现有树形结构，还有很多未整理的 Mod，希望基于现有结构进行分类
- Mod 通常不是单个文件，而是本体 + 汉化 + 预览图等多个文件的组合，需要整体移动

### 期望的解决方向

提供一个可视化的交互式树状目录，能够让用户快速将文件从一处"放置"到树中另一位置。减少移动文件所需的点击步骤。

---

## 问题四：需要借助外部 AI 辅助批量分类

### 问题描述

大量积压的未分类 Mod 需要按照分类体系进行整理。手动逐条处理效率低，希望能够利用外部免费 AI 对话来辅助分类建议。

### 实际痛点

- Mod 积累太多，纯手动处理难以快速完成初次清理
- 不想花钱用 Agent API 在本地直接操作，希望用网页免费 AI 对话来帮忙分类

### 期望的解决方向

- 导出待整理 Mod 文件列表（JSON 格式）和现有分类目录结构（JSON 格式）
- 将 JSON 发送给外部网页 AI 进行分类建议
- 导入 AI 返回的建议 JSON，在本地审阅并确认后再应用
- AI 只生成建议，不直接操作文件系统

### 当前优先级的调整（确认于 2026-07-12）

复盘过程中已确认：AI 辅助可以后置，**先把手动整理体验做舒适**。在暂存区操作、目录树交互、预览图、基础文件操作等工作流完善之前，不优先投入 AI JSON 交换。

---

## 问题五：需要快速搜索受管理内容

### 问题描述

用户希望像 Everything 一样能够快速搜索受管理目录中的 Mod 内容。

### 实际痛点

- 希望搜索功能兼容 UTF-8 编码，中文路径没有任何问题
- 搜索范围是受管理的 Mod 内容，不需要全盘搜索

### 期望的解决方向

在受管理目录范围内提供快速、支持中文的搜索功能，能够搜索中文名、文件名、用途说明等内容。

---

## 问题六：非英语母语用户的中文路径需求

### 问题描述

用户是非英语母语者。现有的 Mod 存储路径全部为英文，阅读效率低。

### 实际痛点

- 英文路径阅读速度比中文慢，中文信息含量高，一目十行读起来快
- 之前被其他 AI 建议使用纯英文路径，但实际使用中英文路径看得累
- 不希望因为支持中文显示而修改实际文件夹路径（因为 MO2 等工具已经基于当前路径工作）

### 期望的解决方向

- 在不改动实际文件夹路径的情况下，能够给文件夹设置中文显示名
- 所有 UI 优先使用中文显示

---

## 原始问题总结表

| 编号 | 问题领域 | 核心痛点 | 期望解决方向 |
|------|---------|---------|------------|
| P1 | 文件夹跳转 | 多层级目录间跳转点击过多 | 交互式树状目录，快速定位 |
| P2 | 预览识别 | 文件名字无法帮助辨认 Mod 内容 | 关联预览图，辅助识别 |
| P3 | 分类操作 | 移动文件到不同文件夹步骤繁琐 | 可视化树状操作，减少点击 |
| P4 | 批量分类 | 积压大量 Mod，逐条处理效率低 | AI JSON 导入导出辅助分类 |
| P5 | 搜索效率 | 无法快速定位已管理内容 | UTF-8 兼容的快速搜索 |
| P6 | 中文支持 | 英文路径阅读效率低 | 不修改实路径，显示中文名 |

## 当前实际工作流

```
用户启动应用
    ↓
添加受管理根目录（一个或多个）
    ↓
手动触发扫描（扫描期间 UI 进入扫描模式）
    ↓
扫描完成后：
  ├─ 左栏目录树显示已扫描目录结构（只读浏览）
  ├─ 中栏素材池显示未关联文件/文件夹
  └─ 右栏显示选中目录的详情
    ↓
用户在素材池选择文件/文件夹
    ↓
用户点击"新建 Mod 条目"（自动关联选中素材）
  或先选中已有 ModItem → 点击"关联到选中条目"
    ↓
在右栏设置成员角色（本体/汉化/预览图/说明/可选文件/未知）
    ↓
在右栏填写元数据（显示名称/说明/来源链接/标签）
    ↓
（可选）设封面：将 preview 角色成员设为封面缩略图
    ↓
（Task 5 未实现：移动到选中目录 → 预演 → 确认 → 执行 → 撤销）
```

## 当前架构的核心抽象

| 概念 | 说明 | 对应原始需求 |
|------|------|------------|
| ManagedRoot | 用户配置的受管理根目录 | 受管理目录范围界定 |
| FolderNode | 扫描得到的目录树节点（存于数据库） | 树状目录（只读） |
| FileAsset | 真实文件或文件夹的索引记录 | 素材条目 |

| OperationLog | 文件操作记录（预演/执行/撤销） | 安全移动保障 |
| ThumbnailCache | 缩略图缓存（文件 + 元数据表） | 预览图展示 |

# 五、当前设计与最初目标的差距

## 原始目标

一个方便的结构清晰的能展现真实文件结构路径的界面工具

## 当前实现

使用虚拟映射，对路径扫描出的文件内容进行一个个手动整理

无视了已经整理好的真实路径结构

## 差距分析

在实现方式上，我的想法是尽量简单真实，之前ai写的想法却是尽量不影响原文件，进行虚拟文件夹分类构建

### 产品方向问题

方向完全偏离，UI内容太重目前，手动分类工作变得繁琐

### 实现问题

即使"直接操作真实文件"的产品方向不变，当前实现中仍有以下问题需要针对性解决：

**UI 与交互层面**
- 封面预览图在当前 UI 中尺寸太小（缩略图生成限制为小尺寸），无法实际用于辨认 Mod 内容
- 素材池缺少文件类型筛选（只看图片/只看压缩包等）、不同排序方式（按文件名/大小/修改时间）和关键词搜索
- 没有复制、粘贴、删除、剪切等基础文件操作
- 工作流步骤过多（扫描 → 素材池 → 创建 ModItem → 关联素材 → 设角色 → 填元数据 → 设封面 → 移动），串联操作不够流畅

**功能性缺失**
- 文件夹中文显示名功能虽已设计（FolderNode.display_name 字段存在），但始终未使用
- 没有"复制文件路径"功能，用户无法快速获取 Mod 文件所在路径供外部工具（如 MO2）使用
- 没有窗口置顶功能，全屏操作时容易被其他窗口遮挡

# 六、现有成果评估

## 建议继续保留

- **文件扫描器（FileScanner）**：递归只读扫描，中文路径支持，扩展名分类，错误容错
- **安全移动与撤销服务（FileOperationService）**：预演、确认、执行、撤销的完整流程，B1/B2 安全校验
- **缩略图生成与缓存（ThumbnailGenerator/ThumbnailCoordinator）**：只读加载，缓存隔离，按需生成
- **数据库层与 Repository**：SQLite schema 设计、迁移机制、CRUD 实现
- **测试体系**：335 项测试覆盖了核心路径，中文路径、安全边界等
- **路径标准化工具（path_utils.py）**：path_key 策略（normcase+normpath）
- **Qt 后台线程模式（ScanWorker/ThumbnailWorker）**：独立连接、信号回传，可作为后续任务的参考模式

### 原因

以上模块是技术基础设施，与"操作真实文件还是虚拟映射"的产品方向无关。无论产品方向如何调整，这些能力都是需要的基础设施。现有实现的质量和测试覆盖可以继续使用。

---

## 建议重新思考

- **ModItem 虚拟映射体系**：当前 ModItem 是脱离文件系统的逻辑层，需要先创建条目、设置角色、填写元数据才能进行操作
- **素材池工作流**：将文件扫描到数据库后再通过"关联到 ModItem"的方式来管理文件，而不是直接在真实目录结构中操作
- **只读目录树的定位**：当前目录树仅用于"选中目标目录"，而用户期望的树是交互式的操作中心
- **三栏式 UI 布局**：当前布局围绕"素材池-条目-详情"展开，与"以目录树为中心"的使用方式存在差异
- **三阶段优先级安排**：基础文件操作、搜索、AI JSON 在当前 roadmap 中分散在阶段 3~5；复盘已确认 AI 辅助可以后置，但基础文件操作（复制/粘贴/移动/拖拽等）和搜索应优先于 AI

### 原因

以上模块体现了"虚拟映射"的设计取向，本质上是"尽量不影响原文件安全"的设计哲学。这与用户"直接操作真实文件"的原始需求存在根本性矛盾。

---

## 暂时无法确定

- **UI 的细节交互方式**（右键菜单、拖拽、复制粘贴等需要桌面文件管理操作）—— 需要先确认产品方向才能决定
- **N 网预览图导入** —— 独立功能，与方向无关，优先级待定
- **AI JSON 的交换格式** —— 需要先确认工作流（是导出整批交给 AI，还是仅导出未分类列表）
- **搜索的具体范围**（是仅搜索已创建的 ModItem，还是搜索文件系统原始内容）
- **文件夹中文显示名的存储方式**（现有 display_name 字段方案是否满足需求）
- **暂存区的具体行为**（是直接使用真实文件夹作为暂存区，还是应用内虚拟暂存）—— 取决于采用"直接操作真实文件"还是"混合"方向

# 七、目前对产品方向的新思考

> 以下为用户当前的理解，基于产品复盘前四个部分的梳理得出。仍可继续调整和细化。

## 当前理解

这个工具真正应该解决的核心问题是：

**提供一个以真实文件系统树状目录为操作中心的 Skyrim Mod 文件管理器 —— 让用户能够直接在树状目录结构中完成文件的分类、移动、预览和整理，而不需要经过虚拟映射步骤。**

简单来说就是：Windows 资源管理器做不到的事情（树状交互好用、有 Mod 预览图），这个工具来做；但应该像资源管理器一样直接操作真实文件，而不是在数据库里建一套虚拟结构。

## 希望达到的工作流程

### 核心流程：暂存区整理 + 目录树操作

```
用户启动应用
    ↓
打开一个现有的 Mod 根目录（暂存区和分类目录均在同一个根目录下）
    ↓
左侧显示目录树（展开/折叠自由操作）
    ↓
用户选中"暂存区"目录（一个真实的文件夹，用来放待整理的 Mod）
    ↓
右侧显示暂存区内的所有文件（图片可直接预览，压缩包显示名称和大小）
    ↓
【方式 A：多文件 Mod 整理】
  ├─ 从暂存区选中本体文件（压缩包/文件夹）
  ├─ 右键 → "新建 Mod 组"
  │   ├─ 自动从文件名剔除后缀和无用信息（如版本号、网址标注等），只保留 Mod 名称
  │   ├─ 以下拉框提供选项：剔除后的纯 Mod 名 / 完整原名（参考 MO2 安装 Mod 时的起名逻辑）
  │   └─ 确认后以所选名字创建同名子文件夹
  ├─ 再选中汉化、预览图等附属文件 → 拖入该子文件夹（实际移动真实文件）
  ├─ 拖入的预览图文件自动重命名为文件夹的名字，便于排序和识别
  └─ 然后将整个子文件夹拖入左侧目录树的目标分类位置（实际移动）
    ↓
【方式 B：单文件 Mod 整理】
  ├─ 选中本体文件
  └─ 直接拖入左侧目录树的目标位置（实际移动单个文件）
    ↓
【方式 C：AI 辅助批量分类】（优先级靠后，先做手动整理）
  ├─ 选中暂存区的一批文件 → 导出 JSON（含文件名/大小/路径）
  ├─ 交给网页 AI 分类建议
  ├─ 导入 AI 建议的 JSON → 审阅 → 确认 → 自动移动到目标目录
  └─ 多文件 Mod：AI 也可建议哪些文件应组合为一组
```

### 其他期望的操作

- **复制文件路径**：在任何文件或目录上右键 → "复制完整路径"，快速获取文件的真实路径字符串（用于 MO2 或其他外部工具）
- **置顶按钮**：窗口右上角或工具栏有一个"置顶"切换按钮，全屏操作时窗口不会被其他程序遮挡
- **右键菜单**：基本的文件操作（复制/剪切/粘贴/删除/重命名），以及"复制路径"、"新建 Mod 组"等特化操作
- **预览图**：Mod 文件旁自动显示关联的预览图（同级目录下的图片或手动指定）；支持缩放

### 方向 C 的完整定义

#### 核心理念

> **真实文件系统是唯一的事实来源（Single Source of Truth）。数据库是元数据增强层，而不是另一套文件组织体系。**

#### 数据库定位

数据库不再是定义"文件属于哪个 ModItem"的组织层，而是记录**真实目录/文件自身无法表达的信息**：

```
真实目录                 数据库（元数据层）
────────────             ────────────────
文件组织结构              → 中文显示名
文件命名                  → 标签（多维度）
目录层级                  → 备注
                          → 来源 URL
                          → 评分 / 是否整理完成
                          → 封面关联
                          → 搜索索引
                          → AI 分析结果
```

数据库里没有"哪些文件属于这个 Mod"这种信息，因为目录本身已经表达了文件归属。

#### 记录对象：内容单元（Content），替代 ModItem + FileAsset

数据库记录的不再是"每个文件"（FileAsset）也不是"虚拟条目"（ModItem），而是**一个真实路径上的内容单元**：

- **大多数时候是一个文件夹**（Mod 组：本体+汉化+预览图在同一目录下）
- **少数时候是一个单文件**（小型 Mod，暂时不需要建文件夹）
- **以后可以是教程文档、截图集、视频项目等**

一条记录示例：

```text
类型：Mod                          # 内容类型（Mod / Tutorial / Artwork / …）
路径：AssetLibrary/Mods/Armor/BDOR  # 真实路径
标题：BDOR Black Knight             # 中文显示名
标签：重甲, 女性, SMP, 4K, 推荐     # 多维标签
来源：https://nexusmods.com/...     # N 网链接
备注：已测试，需刷 BodySlide
评分：★★★★★
封面：preview.webp                  # 封面图片文件名
状态：已整理                        # 整理状态
```

#### 标签替代深层目录嵌套

目录树保持扁平（建议最多二级分类），细分类由数据库标签系统承担：

```text
AssetLibrary/
    Mods/
        Armor/
        Animation/
        NPC/
        ...
    Tutorials/
        BodySlide/
        xEdit/
    Artwork/
        Screenshot/
```

所有深层分类信息（重甲、女性、黑色、HDT、SMP、4K、汉化）都存为标签，通过搜索/筛选替代目录嵌套。

#### 产品定位：从"Skyrim Mod 工作台"到"本地数字资产管理"

产品不再是单纯的 Skyrim Mod 管理器，而是可以管理多类资产的本地 DAM 工具：

- Skyrim Mods（第一阶段重点优化）
- 教程文档（Markdown、PDF）
- 创作素材
- 截图 / 壁纸
- 视频项目
- 等等

各资产类型的核心模型（内容单元 + 路径 + 元数据）是统一的，只是字段和标签体系有所差异。

### 方向 A（已归档，供参考）

曾经考虑的另一个候选路径：不保留数据库元数据层，完全依赖文件系统。经评估后确认不符合长期需求（标签、搜索、多资产类型支持都需要数据库层）。

### 当前设计与方向 C 的差距对比

| 维度 | 当前设计 | 方向 C |
|------|---------|--------|
| 数据库角色 | 第一事实来源，定义文件组织结构 | 增强层，仅记录目录无法表达的信息 |
| 记录对象 | FileAsset（每条文件记录）+ ModItem（虚拟容器） | **内容单元**（一个目录/一个单文件 = 一条记录） |
| 成员管理 | 角色系统（本体/汉化/预览图），需手动分配 | **不需要**——文件已在同一目录中，目录本身表达了关系 |
| 素材池 | 主要工作区，长期维护 | **临时待整理区**，整理完就空 |
| UI 核心 | 三栏布局（根/素材池/详情） | 目录树 + 内容列表 + 元数据面板 |
| 目录树交互 | 只读浏览 | 可操作（拖拽、右键菜单），标签替代深层嵌套 |
| 产品定位 | Skyrim Mod 工作台 | **本地 DAM 工具**，Skyrim 为第一阶段 |

### D1：不为每个 Mod 配备 meta.ini

- **问题**：是否需要像 MO2 一样为每个 Mod 文件夹创建一个 meta.ini 文件，用于记录 Mod 路径、N 网原址、描述等信息？
- **决策**：不做。理由：
  - 本工具直接操作真实文件，文件路径本身就是信息来源，不需要额外记录
  - N 网尾号可从压缩包文件名提取
  - 描述等元数据在当前工作流中无实际使用场景
  - 在目录中写额外文件会增加管理负担，且对目录迁移不友好
- **影响**：保持纯文件系统操作，不引入元数据文件写入

### D2：选择方向 C 作为产品方向

- **问题**：在产品复盘的 A（真实目录唯一，无数据库）和 C（数据库作为元数据增强层）之间选择哪个方向？
- **决策**：选择方向 C。理由：
  - 真实目录难以独立表达标签、评分、来源、备注等信息，需要数据库作为增强层
  - 方向 C 可服务于更广泛的资产类型（Mod、教程、截图、视频项目等），长期扩展性好
  - "内容单元"（Content）替代 ModItem + FileAsset 的概念已被充分讨论和确认
  - 现有代码资产在方向 C 下可复用率更高（约 2200 行可保留）
- **影响**：后续 spec.md、architecture.md、roadmap.md 需围绕"内容单元 + 路径元数据"模型重写

# 九、结论

## 当前结论

经过产品复盘，可以明确以下几点：

1. **原始需求是清晰的**：用户需要一个能直接操作真实目录结构的本地资产管理工具，核心是树状交互 + 元数据标记 + 预览图 + AI 辅助
2. **当前实现出现了方向偏移**：在开发过程中逐渐走向了"虚拟映射 + ModItem 体系"的设计路径
3. **偏移的根源**：AI 助手在设计阶段天然倾向于"安全抽象"——引入虚拟层来保护文件系统，但忽视了用户"直接操作真实文件"的原始需求
4. **最终方向已确定**：方向 C——数据库作为元数据增强层，真实目录为唯一事实来源；记录对象为"内容单元"而非文件或虚拟条目
5. **有可复用的技术资产**：扫描器、安全移动、缩略图、数据库等基础设施质量良好，在新的方向下大部分可复用或改造后复用

## 下一步计划

2. ✅ 已完成 A/C 方向代码复用评估（见 Q2）
3. ✅ **最终方向已确认：方向 C**
4. **⏳ 更新设计文档**：根据方向 C 的定义，重写 spec.md、architecture.md、roadmap.md
5. **调整开发计划**：以"内容单元"为核心模型重新规划阶段安排，优先级为：**舒适的手动整理体验**（暂存区操作、目录树交互、拖拽移动、预览图） → **元数据标记与搜索** → **AI 辅助**
