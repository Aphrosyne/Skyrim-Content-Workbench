# 阶段 2 计划：基础桌面工作台

## 1. 唯一核心目标与完成定义

### 唯一核心目标

让用户能够在桌面应用中完成一次真实、可预演、可恢复的本地 Mod 整理流程：

> 配置受管理根目录 → 扫描并建立素材池 → 手动创建 ModItem 并关联多个文件/文件夹 → 在目录树中选择目标分类 → 预演并确认移动 → 查看执行结果与可撤销操作。

阶段 2 不追求完整内容管理器，不实现搜索、AI 分类、联网预览图导入或复杂拖拽；重点是将阶段 1 已有 API 变成一个稳定、不会绕过文件安全边界的可操作 GUI。

### 完成定义

阶段 2 完成时，用户可以在 Skyrim Content Workbench 中：

1. 添加、查看和移除受管理根目录配置。
2. 手动触发扫描，并在 UI 中看到扫描结果、错误摘要和未关联素材。
3. 在素材池中选择文件或文件夹，创建 ModItem，并为成员指定现有角色。
4. 在树状目录视图中浏览已扫描目录，并选择一个目标分类目录。
5. 对一个 ModItem 发起“移动到此目录”，先看到逐成员预演结果，再明确确认执行。
6. 在执行结果中看到成功、失败和阻止原因；对已完成且安全的操作发起撤销预演并确认撤销。
7. 所有用户文件位置变化仍只由 `FileOperationService` 执行；UI 不直接使用 `rename`、`shutil`、删除或覆盖逻辑。
8. 中文路径、中文显示名和 UTF-8 数据在扫描、展示、组装、移动与撤销流程中保持可用。

阶段 2 的验收不要求卡片式图库、缩略图生成、文件拖拽移动、全文搜索或 AI JSON。

## 2. 明确不做的内容

阶段 2 不做以下内容，避免把基础工作台扩展成多个并行项目：

* 不实现 SQLite FTS5、Everything 式快速搜索、标签筛选或全文搜索。
* 不实现 AI JSON 导入、导出、候选分组、自动分类或任何 AI 调用。
* 不实现联网访问、Nexus 页面解析、预览图下载、网页爬取或账号功能。
* 不实现图片缩略图生成、预览图墙、联系表或封面自动选择。
* 不实现从 Windows 资源管理器拖入应用。
* 不实现卡片拖入目录树、树内拖拽移动或文件系统拖放。
* 不实现自动分组、按文件名推断本体/汉化/预览图关系。
* 不实现文件监听、实时增量索引、后台持续扫描。
* 不实现压缩包内容解析、MO2 直接集成、安装或加载顺序管理。
* 不实现跨设备同步、云端数据、账号、多人协作。
* 不实现打包发布、自动更新或安装程序。
* 不修改用户文件内容，不删除用户文件，不覆盖同名目标文件。

## 3. 阶段 2 开始前必须确认的决策

以下问题会直接改变 UI 结构或数据迁移；应先更新 `docs/open-questions.md`，再开始对应任务。

### D1：受管理根目录的最小数据模型

确认是否新增独立的 `managed_root` 表，而不是仅依赖 `folder_node.is_managed_root`。

建议决策：新增 `managed_root` 表，至少包含 `id`、`real_path`、`path_key`、`display_name`、`created_at`、`updated_at`。扫描得到的 `FolderNode` 继续表示目录树；根目录配置则独立保存，避免扫描结果被清理或重建时丢失用户配置。

这会影响数据库 schema v2、迁移和根目录设置 UI。

### D2：阶段 2 的移动入口

确认采用“仅按钮式移动”还是“按钮 + 右键菜单”。

建议决策：阶段 2 仅提供明确按钮入口，例如“移动到选中目录”，可选增加右键菜单；不实现拖拽。这样预演、确认和错误展示更容易保持一致。

这会关闭 Q3 的阶段 2 部分，并将拖拽留到后续版本重新评估。

### D3：未关联素材的最小处置方式

确认阶段 2 是否只展示未关联素材，还是允许“忽略”。

建议决策：仅展示，不增加忽略状态、不移动、不删除。用户可以把素材关联到 ModItem；未关联内容继续保留在素材池。

这会保持 Q11 未关闭，但避免为了 UI 引入新的状态模型。

### D4：扫描执行模型

确认阶段 2 是否采用后台线程包装现有同步扫描器。

建议决策：使用 Qt 后台工作线程包装 `FileScanner.scan_many()`，提供进度文本和完成/失败状态；取消功能不在阶段 2 承诺范围内。不得修改 `FileScanner` 的同步接口。

这会部分回答 Q18，但不引入复杂并发、取消或文件监听。

### D5：ModItem.status

建议决策：阶段 2 不新增 `ModItem.status`。列表展示和可操作性由是否有关联成员、是否设置分类目录、是否有封面等现有字段表达，不做“草稿/完成/归档”筛选。

这将关闭 Q1，避免不必要的 schema 迁移和 UI 状态设计。

### D6：成员角色限制

建议决策：保持当前 `MAIN_MOD≤1`、`README≤1`、其余不限的实现，不在阶段 2 改动。

这将关闭 Q19，UI 只需把服务层返回的限制错误展示给用户。

### D7：部分失败的 UI 处理

建议决策：保持当前“不自动回滚”。执行结果必须明确列出成功成员、失败成员和可撤销范围；用户可对已成功部分执行撤销预演。

这将关闭 Q20 的阶段 2 决策，避免在跨盘移动中增加自动回滚风险。

## 5. 原子任务详细说明

### 任务 1：受管理根目录持久化与 schema v2

#### 目标

为用户选择和保存受管理根目录提供稳定的数据模型与应用服务，使应用重启后仍知道需要扫描哪些目录。

#### 输入

* 当前 `app.db` schema v1。
* 现有 `FolderNode`、`FileScanner.scan_many()` 与 `is_managed_root` 字段。
* 用户通过未来 UI 选择的本地目录路径。

#### 输出

* schema v2 迁移。
* `ManagedRoot` 领域模型。
* `ManagedRootRepository`。
* `ManagedRootService`，负责添加、列出、移除和校验根目录。
* 不修改用户目录、不自动扫描。

#### 涉及模块

* `src/domain/models.py`
* `src/infrastructure/migrations.py`
* `src/infrastructure/repositories/managed_root.py`
* `src/infrastructure/repositories/errors.py`
* `src/application/managed_root_service.py`
* `src/application/errors.py`
* 新增测试文件。

#### 验收标准

* 旧 v1 数据库可迁移到 v2，已有业务数据不丢失。
* 可添加存在且为目录的根目录。
* 不允许重复 `path_key` 根目录。
* 不允许添加不存在路径或普通文件路径。
* 可列出根目录，中文路径往返正确。
* 可移除根目录配置；移除配置不删除、不移动、不修改该目录及其中任何用户文件。
* 该任务不要求删除对应 `FolderNode` 扫描记录；其清理策略保持待确认。

#### 测试要求

* v1→v2 迁移测试。
* Repository CRUD、唯一约束、中文路径测试。
* Service 对不存在路径、文件路径、重复路径的测试。
* 移除根目录后真实目录与文件内容、mtime、size 不变的测试。

#### 文件安全约束

* 仅写应用数据库。
* 不扫描、不移动、不删除、不重命名用户文件。
* 路径检查只能使用只读文件系统 API。

#### 范围外内容

* 根目录选择对话框。
* 扫描触发。
* 根目录重命名、排序、标签。
* 自动清理历史扫描数据。

### 任务 2：扫描工作流应用层与后台任务适配

#### 目标

将现有同步扫描器包装为可被 GUI 调用的扫描工作流，扫描受管理根目录并安全持久化结果，不阻塞 Qt 主线程。

#### 输入

* 任务 1 的 `ManagedRootService`。
* 现有 `FileScanner` 与 `persist_scan_result()`。
* 现有 FolderNode、FileAsset repository。

#### 输出

* `ScanWorkflowService`：读取已配置根目录，执行扫描与持久化，返回结构化摘要。
* Qt 适配层或 worker：在后台线程运行扫描工作流，并向 UI 发送开始、进度文本、完成、失败事件。
* 扫描摘要至少包含：根目录数、扫描条目数、持久化条目数、错误列表或错误计数。

#### 涉及模块

* `src/application/scan_workflow_service.py`
* `src/application/errors.py`
* `src/infrastructure/file_scanner.py`
* `src/app/` 下新增 Qt worker 或 controller 适配模块。
* 相关 repository 与测试。

#### 验收标准

* 无根目录时，工作流返回用户可展示的空状态，而不是异常崩溃。
* 多个根目录可被依次扫描。
* 单个根目录扫描失败不会阻止其他根目录扫描。
* 扫描结果能写入现有 FolderNode 与 FileAsset。
* 同一路径被多个根目录覆盖时仍按既有 `path_key` 去重策略处理。
* 后台扫描期间 Qt 主线程仍可响应最基本事件。
* 扫描完成后 UI 调用方可刷新目录树、素材池和 ModItem 相关视图。

#### 测试要求

* 使用临时目录的完整扫描工作流测试。
* 多根目录、中文路径、重叠目录、扫描错误继续执行测试。
* 无根目录测试。
* worker 层至少测试信号或回调会返回成功与失败摘要；不依赖真实用户目录。
* 保留现有只读扫描保证测试。

#### 文件安全约束

* 仅调用现有只读扫描 API。
* 不得在 worker、controller 或 UI 中写用户文件。
* 持久化仅写应用数据库。

#### 范围外内容

* 扫描取消。
* 文件监听。
* 增量扫描。
* 扫描进度百分比的精确统计。
* 自动关联、自动分类。

### 任务 3：主窗口骨架与目录树、素材池、ModItem 列表

#### 目标

将空白主窗口替换为最小可浏览工作台，让用户看见受管理根目录、扫描目录树、未关联素材和已有 ModItem。

#### 输入

* 任务 1 的根目录配置服务。
* 任务 2 的扫描工作流。
* `FolderNodeRepository`、`FileAssetRepository`、`ModItemRepository`。
* 当前空白 `MainWindow`。

#### 输出

* 三栏或等价固定布局：

  * 左侧：受管理根目录与目录树。
  * 中间：未关联素材池。
  * 右侧：ModItem 列表与基础详情区域。
* 根目录添加、移除、扫描按钮。
* 状态栏或状态区域，用于展示扫描摘要与错误提示。
* 目录树以数据库中的 `FolderNode` 为数据源，不直接把文件系统树作为可写模型。

#### 涉及模块

* `src/app/main_window.py`
* `src/app/` 下 UI controller、view model 或 widgets。
* 可新增 `src/application/workbench_query_service.py`，集中 UI 查询逻辑。
* 现有 repositories 与测试。

#### 验收标准

* 首次启动无根目录时显示明确空状态。
* 可通过目录选择对话框添加根目录，并在左侧显示。
* 点击扫描后，树状视图显示已持久化目录层级。
* 素材池显示未关联 `FileAsset`，至少展示文件名、真实路径、类型提示或角色。
* ModItem 列表显示已有条目的 display name 和基础成员数量。
* 中文目录、中文文件名、中文 ModItem 名称显示正确。
* UI 刷新不直接调用文件移动 API，不直接修改用户文件。
* 目录树仅用于浏览和选择；本任务不实现拖放。

#### 测试要求

* MainWindow 构造测试升级为：无根目录、已有根目录、已有扫描数据、中文数据。
* 查询服务测试覆盖树构建、未关联素材查询、ModItem 列表查询。
* Qt UI 测试可使用临时 SQLite 与临时目录，不依赖真实 `%LOCALAPPDATA%`。
* 回归运行全部阶段 1 测试。

#### 文件安全约束

* UI 只读取数据库与只读路径信息。
* 添加或移除根目录只修改应用数据库。
* 扫描必须通过任务 2 的工作流。
* 不得在 widget 事件中调用 `Path.rename`、`shutil`、`unlink` 或等价 API。

#### 范围外内容

* 卡片图库。
* 缩略图。
* 搜索栏。
* 文件或卡片拖拽。
* ModItem 创建和编辑表单。
* 移动与撤销操作。

* 从素材池选择一个或多个未关联素材并关联到 ModItem。
* 成员角色选择与修改。
* 移除成员。
* 编辑 display_name、description、source_url、category_folder_id、tags。

#### 涉及模块

#### 验收标准

* 不移动、不复制、不删除、不重命名任何用户文件。

### 任务 5：安全移动与撤销确认工作流 UI

#### 目标

将现有 `FileOperationService` 接入明确的预演、确认、执行、结果和撤销流程，使用户可以从 ModItem 整理到目录树目标位置，同时保留阶段 1 的全部安全保证。

#### 输入

* 任务 3 的目录树目标选择。
* 任务 4 的 ModItem 详情与成员数据。
* 现有 `FileOperationService`、`OperationLogRepository`、MovePlan、UndoPlan。

#### 输出

* “移动到选中目录”入口。
* MovePlan 预演对话框：逐成员显示源路径、目标路径、跨盘提示、阻止原因和是否可执行。
* 明确确认按钮；不可执行时确认按钮禁用。
* 执行结果面板：成功、失败、失败原因、可撤销状态。
* 最近操作列表或最小操作记录视图。
* 撤销预演对话框与明确确认撤销入口。

#### 涉及模块

* `src/app/main_window.py`
* 新增 move/undo dialogs、controllers、view models。
* `src/application/` 可新增 UI 适配用 facade，但不得复制文件操作规则。
* `src/infrastructure/file_operation_service.py` 仅在发现 UI 所需且不改变产品决策的缺失查询接口时最小扩展。
* `OperationLogRepository` 与测试。

#### 验收标准

* 用户必须先选择一个 ModItem 和一个目录树目标，才可发起移动预演。
* UI 只调用 `plan_move()` 获取预演；不得自行计算或执行移动。
* 重名、源不存在、目标不可写、目标为自身或子目录等阻止原因可在 UI 中展示。
* 只有用户明确确认后才调用 `execute_move()`。
* 执行后 UI 刷新受影响 ModItem、目录树、素材池和操作记录。
* 部分失败时明确显示成功与失败成员；不显示“全部成功”。
* 用户可对符合条件的已完成操作发起 `plan_undo()`；不安全时展示阻止原因且不得调用 `execute_undo()`。
* 用户明确确认撤销后才调用 `execute_undo()`。
* 不允许覆盖目标同名文件；UI 不提供“强制覆盖”选项。

#### 测试要求

* UI/controller 集成测试覆盖完整 happy path：预演→确认→执行→刷新→撤销预演→确认撤销。
* 覆盖取消确认后不执行移动。
* 覆盖不可执行预演时确认按钮不可用。
* 覆盖部分失败结果展示。
* 覆盖 B1/B2 撤销阻止结果展示。
* 保留并运行阶段 1 的 23 个文件操作服务测试。
* 所有真实文件测试仅使用 `tmp_path`。

#### 文件安全约束

* UI 不得直接访问写文件 API。
* 所有移动与撤销必须经 `FileOperationService`。
* 不增加删除、覆盖、自动回滚或批量无确认执行。
* 操作前必须存在持久化 `OperationLog(planned)`。
* 跨盘行为保持既有 `copy2 + unlink` 移动语义，不在 UI 层自行实现。

#### 范围外内容

* 卡片拖入目录树。
* 从资源管理器拖入。
* 批量移动多个 ModItem。
* 自动回滚。
* 自定义冲突策略。
* 重命名、复制、删除操作。
* 操作历史导出。
