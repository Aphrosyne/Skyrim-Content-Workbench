# Skyrim Mod Workbench — 产品规格说明

## 1. 项目定位

Skyrim Mod Workbench 是一个面向 Windows 的、本地优先（local-first）的 Skyrim Mod 整理工作台。

它不替代 Mod Organizer 2，也不替代 Everything 或 Windows 资源管理器。它解决的核心问题是：用户积累了大量尚未分类的 Mod 文件，且一个逻辑 Mod 往往由本体、汉化、预览图、说明等多个文件组成；用户需要通过预览、中文信息和可视化目录树，快速完成识别、分组、分类与归档。

项目未来计划开源，第一版不包含云同步、账号系统或用户数据上传。

## 2. 核心原则

* local-first：所有数据默认保存在本机；不上传用户 Mod 文件、路径、图片或元数据。
* 文件系统是真实来源：Mod 文件始终是普通文件，能够脱离本程序继续被资源管理器、MO2 和其他工具使用。
* 数据库是管理层：数据库仅保存索引、中文信息、逻辑分组、预览关联、分类映射和操作日志。
* 安全优先：所有涉及用户 Mod 文件的操作必须可预演、可确认、可记录、可恢复。
* 人工确认优先：AI 只能生成建议，不能直接执行文件移动或修改。
* 中文优先：真实路径、中文目录、中文显示名、中文搜索均应正常工作。
* 不依赖统一命名规则：不得假设 Nexus Mods、本地汉化包、社区分享文件或预览图存在可靠命名规律。

## 3. 第一版目标

第一版的目标是帮助用户清理既有的大量未分类 Mod：

1. 扫描用户指定的 Mod 根目录和待整理目录。
2. 将压缩包、文件夹、图片、文档等文件显示为待处理素材。
3. 由用户手动创建“Mod 条目”，并把多个文件关联为一个逻辑组。
4. 为 Mod 条目填写中文名称、一句话用途说明、来源和分类。
5. 为 Mod 条目设置本地预览图作为封面。
6. 在可视化目录树中选择目标分类目录。
7. 将 Mod 条目的所有成员文件安全地整体移动到真实目标目录。
8. 导出待处理条目或未归类素材清单为 JSON，交给外部网页 AI 分类。
9. 导入 AI 返回的建议 JSON，在本地审阅并确认后应用。
10. 按中文名、原文件名、用途、分类和标签搜索已管理内容。

## 4. 非目标

第一版不包含以下功能：

* 云同步、账号、多人协作或远程数据库。
* 替代 MO2 的安装、排序、启用、冲突管理或游戏启动功能。
* 替代 Everything 的全盘文件搜索。
* 自动可靠识别 Mod 本体、汉化、预览图之间的关系。
* 不经用户确认的自动移动、删除、覆盖或重命名。
* 从 Nexus Mods 自动批量抓取预览图。
* 解析压缩包内部图片。
* 从应用向 Windows 资源管理器拖出真实文件。
* 文件哈希去重、完整文件监听、MO2 深度联动。
* 本地 AI 推理或 AI 直接控制文件系统。

## 5. 用户工作流

### 5.1 清库存主流程

1. 用户选择一个或多个本地根目录。
2. 程序扫描并建立索引。
3. 用户在“待处理素材池”中查看压缩包、图片、文档和文件夹。
4. 用户选中一个或多个素材，创建一个 Mod 条目。
5. 用户为成员指定角色：本体、汉化、预览、说明、可选文件或未知。
6. 用户填写中文名称、用途说明、来源和分类。
7. 用户设置一张预览图为封面。
8. 用户在左侧真实目录树中选择目标目录。
9. 程序先展示移动预演结果。
10. 用户确认后，程序移动该 Mod 条目的全部成员。
11. 程序写入操作日志，并提供撤销入口。

### 5.2 AI 建议流程

1. 用户选择待处理 Mod 条目或未归类素材。
2. 程序导出 JSON。
3. 用户将 JSON 交给外部网页 AI。
4. AI 返回符合项目 JSON Schema 的建议文件。
5. 用户导入建议文件。
6. 程序显示建议差异，包括中文名、用途、分类、标签和候选成员关系。
7. 用户逐项或批量勾选确认。
8. 文件移动仍必须经过预演和确认。

## 6. 数据对象

### 6.1 ModItem

逻辑上的一个 Mod 条目，可包含多个真实文件。

字段：

* id
* display_name：中文显示名，可为空
* description：一句话用途说明，可为空
* source_url：来源链接，可为空
* category_folder_id：目标分类目录，可为空
* tags：标签集合
* status：待确认，第一版是否需要独立状态字段
* cover_asset_id：封面预览图，可为空
* created_at
* updated_at

### 6.2 FileAsset

一个真实文件或文件夹。

字段：

* id
* mod_item_id：可为空
* real_path
* filename
* extension
* asset_kind：file 或 folder
* role：main_mod、translation、preview、readme、optional_file、unknown
* size_bytes
* modified_at
* imported_at
* batch_id：待确认，第一版是否实现下载批次概念

### 6.3 FolderNode

一个受管理的真实目录节点。

字段：

* id
* real_path
* parent_id
* display_name：可为空；若为空则显示真实目录名
* is_managed_root
* created_at
* updated_at

### 6.4 OperationLog

所有会影响真实文件的操作记录。

字段：

* id
* operation_type
* status：planned、confirmed、completed、failed、undone
* affected_asset_ids
* source_paths
* target_paths
* conflict_policy
* created_at
* completed_at
* undo_payload
* error_message

## 7. 文件操作安全要求

所有文件移动必须遵守以下规则：

1. 前端不得直接操作文件系统。
2. 所有移动必须由统一的文件操作服务执行。
3. 执行前必须生成预演结果。
4. 预演必须列出每个源路径、目标路径、是否跨盘、是否重名、是否缺失、是否不可写。
5. 用户必须显式确认预演结果。
6. 默认冲突策略为“询问”；不得默认覆盖。
7. 禁止把文件夹移动到其自身或子目录。
8. 同盘移动优先使用原子移动或重命名。
9. 跨盘移动必须明确提示其为复制后删除流程。
10. 每个成功移动必须记录源路径和目标路径。
11. 撤销前必须再次检查当前文件状态；不安全时不得强行回退。
12. 单个成员移动失败时，不得将整个 Mod 条目标记为完全成功。
13. 不得删除用户文件；第一版不提供删除功能。
14. 不得修改用户文件内容。
15. 所有预览缓存必须与用户 Mod 文件目录分离。

## 8. UI 结构

主窗口采用三栏布局：

* 左侧：真实目录树。显示受管理根目录及其子目录，可展开、折叠、搜索；支持中文真实目录和中文显示名。
* 中间：待处理素材池、Mod 卡片列表或搜索结果。Mod 条目以卡片显示，优先展示封面预览。
* 右侧：详情编辑区。显示成员文件、角色、中文名、用途、来源、分类、预览图和操作预演。

待确认：第一版是否实现“卡片拖入目录树”作为唯一移动入口，或同时提供“移动到…”按钮。

## 9. 搜索

第一版搜索范围：

* Mod 条目中文名
* 原始文件名
* 用途说明
* 来源链接
* 标签
* 真实路径
* 分类目录显示路径

搜索仅针对受管理目录和本地数据库，不承担全盘搜索职责。

## 10. 预览图

第一版预览图要求：

* 支持将本地图片关联到 Mod 条目。
* 支持从素材池选择图片作为封面。
* 支持常见图片格式，至少包括 JPG、PNG、WEBP。
* 预览图可以是 Mod 条目的真实成员，也可以是外部关联图。
* 应用生成的缩略图缓存必须保存到应用数据目录。
* 不得修改原始图片文件。

待确认：第一版是否允许“外部关联图”不复制到应用缓存，只保存原路径。

## 11. AI JSON 交换

AI 仅处理摘要数据，不获得本地文件操作权限。

导出内容至少包括：

* Mod 条目 ID 或素材 ID
* 文件名
* 扩展名
* 文件角色
* 当前分类
* 中文名和用途（如已有）
* 是否有预览
* 来源链接
* 可选的分类目录列表

导入建议至少支持：

* 建议中文名
* 建议用途说明
* 建议分类目录 ID
* 建议标签
* 候选成员关系
* 置信度
* 原因说明

待确认：是否在第一版导出预览图缩略图联系表；这会提高 AI 判断能力，但增加导出复杂度。
