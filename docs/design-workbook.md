## 讨论一：内容单元（Content）的定义

*状态：✅ 已完成*

### 结论

#### 1. 一个内容单元是什么

采用 C) 混合模式：

- **大多数情况**：一个文件夹就是一个内容单元（Mod 组：本体 + 汉化 + 预览图等文件都在同一目录下）
- **少数情况**：一个单文件也可以独立作为一个内容单元（小型 Mod，暂时不需要文件夹包裹）

#### 2. 内容单元的基本属性

| 属性 | 说明 |
|------|------|
| 路径（唯一标识） | 文件夹路径或文件路径 |
| 内容类型 | Mod / Tutorial / Artwork / …… |
| 标题（中文显示名） | 用户可自定义 |
| 标签 | 多维度标签（见下文"作者"说明） |
| 备注 | 用户自由填写 |
| 来源 URL | N 网链接等 |
| 评分 | 用户评分 |
| 封面 | 指向目录内或外部的图片 |
| 整理状态 | 未整理 / 已整理 / 待复查 |

**关于"作者"**：用户希望将作者信息纳入管理（因为部分作者的作品需要归类在一起）。待讨论：作者是作为独立字段，还是通过标签（如 `作者:Kirax`）表达。

#### 3. Mod 组 = 一个文件夹内容单元

不需要 ModItem + 成员角色体系。一个文件夹里有 main.7z + cn.7z + preview.webp：

- 文件夹本身 → 内容单元
- 类型 → Mod
- 文件已在同一目录下，目录本身表达了关系
- 预览图 → 标记为封面
- **不需要角色系统**

#### 4. 单文件 → 文件夹的升级路径

对于单文件内容单元，提供右键功能：

1. 用户选中单个文件 → 右键 → "创建 Mod 组"
2. 自动从文件名提取主要名称（剔除版本号、无用后缀等，参考 MO2 命名逻辑）
3. 在文件所在目录创建同名子文件夹
4. 将文件移动进去
5. 该文件夹成为新的内容单元

#### 5. 用户的顾虑

- 当前本地存储结构需要先小幅重构（减少树深度），再开始使用新工具扫描
- 自动将文件夹识别为内容单元时，需要**扫描深度限制**，避免嵌套太深的无意义内容被错误识别
- 扫描策略（讨论五）需要与 content 识别联动

### 展开讨论

#### 作者字段处理

- ❌ 排除 C（作者作为目录层级）
- 选定 **B（标签）**：作者信息作为标签之一（如 `作者:Kirax`）
- 理由：N 网下载的 Mod 文件名不附带作者，能记住作者的属于少数特殊情况；大量群友分享的 Mod 来源不明，无法知道作者。因此不适合作为独立字段。

#### 扫描深度与内容单元识别

- 采用 **D + C 混合**：
  - 扫描到指定深度后，**叶子节点（无子文件夹的目录）自动候选为内容单元**
  - 用户可**手动确认/调整**候选结果
  - 非叶子节点（尚有子分类的目录）不自动视为内容单元
- **深远影响**：由于标签系统可以替代深层目录的细分类功能，用户可以借此机会大幅缩减现有目录树的深度：
  - 之前的 `Armor/Heavy/Female/Black/HDT/BDOR/` → 可以简化为 `Armor/BDOR/`，细分类信息存入标签
  - 这正好解决了"树太深"的原始痛点（问题 P1/P3）

## 讨论二：数据库保存什么 / 不保存什么

*状态：✅ 部分确认，标签系统需进一步讨论*

### 结论

#### 1. 数据库保存的内容

| 数据 | 说明 |
|------|------|
| `content_unit` 表 | 内容单元核心数据：路径、类型、标题、来源 URL、评分、封面、整理状态 |
| 标签系统 | 见下方"待进一步讨论" |
| 备注/笔记 | 自由文本，挂在 content_unit 上 |
| `managed_root` 表 | 保留（已有） |
| `folder_node` 表 | **保留但简化**——作为目录树性能缓存，数据源始终是真实文件系统，缓存仅用于加速显示。若后续发现维护成本高于收益，再移除 |
| `thumbnail_cache` 表 | 保留（已有） |
| 操作历史 | **简化版历史记录**，类似 PS 历史记录，统一的一张表，不用每个内容单元独立建表 |

#### 2. 数据库绝对不保存的内容

- 文件的组织关系（不记录"哪些文件属于哪个内容单元"——目录本身已表达）
- 文件路径与内容单元的映射不由数据库定义
- 可随时从文件系统重建的缓存信息

#### 3. 操作历史（简化版）

##### 设计原则

- 不再需要当前的 OperationLog 预演/确认/执行/撤销四步状态机
- 改为一张统一的**操作历史表**，类似 PS 历史记录
- 每次用户执行一个文件操作，直接写一条记录，不搞预演和确认两步
- 撤销时检查"文件还在不在、有没有被改过"，安全就直接还原

##### 数据结构

```text
operation_history
  - id
  - operation_type: move / delete / rename / new_folder
  - source_path
  - target_path（移动时才有）
  - created_at
  - can_undo（系统自动判断）
```

##### 删除策略

- 删除文件/文件夹时 → **移到 Windows 回收站**（利用系统原生回收站）
- 不搞数据库标记删除、也不搞关闭时自动清理
- 用户可以在资源管理器的回收站里找回文件，最安全也最简单
- 操作历史中记录"删除"操作，但实际撤销能力由回收站本身提供

#### 4. 标签系统设计

##### 设计模型

```
标签分类（TagCategory）              标签（Tag）
───────────────                      ──────────
服装护甲（蓝色系）                    重甲, 轻甲, 法袍, 现代, 幻想
武器（橙色系）                        单手剑, 双手剑, 弓, 法杖
作者（绿色系）                        Kirax, NINI, 3HY, …
来源                                  N网, 群友分享, 自购
状态                                 已测试, 已汉化, 待测试, 推荐
部位                                 身体, 头部, 腿部
```

- 一个标签**只属于一个分类**（一对一归属，不跨分类）
- 分类是标签的"分组容器"，用于 UI 筛选时折叠/展开
- 颜色定义在分类层面，同一分类下的标签颜色相近（方便视觉快速辨别）

##### 标签的创建

| 方式 | 说明 |
|------|------|
| **预制**（B） | 预置一套 Skyrim 常用标签体系，用户可增删 |
| **自定义** | 用户打字输入新标签，自动补全已有标签 |
| **AI 扫描辅助**（C） | 可选：在预制基础上，AI 扫描现有目录结构提取候选标签，用户在预制基础上增删 |

初始采用 **B（预制）+ AI 辅助扫描建议** 的组合策略。

##### UI 交互方式

- 筛选时：**先点标签分类 → 显示该分类下的标签 → 多选高亮标签进行筛选**
- 打标签时：打字自动补全 + 最近常用置顶
- 批量操作：选中多个内容单元 → 一次性添加/移除标签
- 标签分类的角色与**目录树一级分类**的作用一致（都是"先选大类，再看具体内容"）

##### 数据结构（概念层）

```text
tag_category
  - id, name, color_hue

tag
  - id, name, category_id

content_unit_tag
  - content_unit_id, tag_id
```

#### 2. 暂存区设计（待定细节）

**暂存区不是固定的一个目录**，用户可以选择任意位置作为暂存区：
- 可以是 `AssetLibrary/_staging/`
- 也可以是 `D:/Downloads/` 或其他任意散落的位置
- 用户把新下载的 Mod 放在哪里，就在软件里把那个目录设为暂存区

**暂存区操作流程：**
1. 在目录树中选中一个暂存目录（或通过"设为暂存区"按钮指定）
2. 内容列表显示该目录下的所有文件
3. 选中一个本体文件 → 右键"创建 Mod 组" → 自动以文件名创建文件夹 → 文件移入
4. 再选中汉化、预览图等 → 拖入该文件夹（或通过按钮关联）
5. 在目录树中选中目标分类目录 → "快速插入"按钮 → 文件夹整体移至目标位置

**待定：创建 Mod 组时的"装配状态"如何展示？**
- 选项 A：直接在目录树中展开暂存区，看到新建的文件夹，双击进入查看内容、拖入更多文件
- 选项 B：弹出一个单独的"装配窗口"或侧边面板，专门展示正在组装的 Mod 内容
- 选项 C：在右栏元数据面板中临时展示当前文件夹的内容

- ✅ 头部标题栏和目录树是必须的

- ✅ 搜索框（全局搜索）
- ✅ 快速插入按钮（整理模式下，选中目标目录 → 一键插入）
- ✅ 右键菜单：复制路径、创建 Mod 组（暂存区）、重命名、删除（移至回收站）
- ✅ 元数据编辑面板：方便操作
- ✅ 大图预览是核心需求（浏览模式下，有预览图就显示大图，没有就显示详细列表）
- ✅ 标签筛选栏（浏览模式下可用）
- ✅ 暂存区不固定，用户可以标记任意目录为暂存区（可以在目录树上看到标记）

## 讨论四：文件移动原则

*状态：✅ 已完成*

### 结论

#### 1. 移动场景与操作方式

| 场景 | 操作方式 |
|------|---------|
| 单文件 → 创建 Mod 组 | 右键 → "创建 Mod 组"，自动提取名称、建文件夹、移入文件 |
| 暂存区附加文件拖入 Mod 组 | 拖拽到装配面板 |
| 暂存区 → 分类目录 | 拖拽到树节点，或选中目标后点「快速插入」按钮 |
| 分类目录间重新分类 | 拖拽到另一个树节点 |
| 单文件升级为 Mod 组 | 右键 → "创建 Mod 组"（同目录建文件夹并移入） |
| 删除 | 移到回收站 |
| 重命名 | 右键重命名 |

操作对象均为**单个文件或单个文件夹**，不再涉及多成员批量移动。

#### 2. 安全规则

| 规则 | 策略 |
|------|------|
| 节点间移动 | **弹出确认提示**，防止误操作影响整体目录结构 |
| 重名冲突 | **弹窗让用户选择**：覆盖 / 跳过 / 重命名（C） |
| 跨盘移动 | **提示"这是跨盘操作"** 后再执行（B） |
| 文件夹移到自身或子目录 | **保留阻止规则**（这操作本身不合理） |

#### 3. 撤销

- 通过简化版操作历史表实现
- 撤销时检查文件状态是否安全
- 删除操作由 Windows 回收站提供撤销能力

## 讨论五：扫描策略

*状态：✅ 已完成*

### 结论

#### 1. 触发方式：C（启动时自动 + 手动补扫）

- 软件启动时自动扫描，确保目录树缓存和内容单元候选是最新的
- 用户也可以通过按钮手动触发重新扫描
- 原因：用户可能在不开软件时修改/增加目录，自动扫描避免遗忘

#### 2. 增量扫描：基于目录 mtime

- 启动时检查每个目录的 mtime（修改时间）与上次扫描时记录的值
- 相等 → 跳过，沿用缓存
- 不等 → 重新扫描该目录及子目录
- 不是绝对保证（某些操作可能不更新 mtime），用户可手动全量重扫作为兜底
- 不做实时文件系统监听（避免 CPU 负载影响游戏）

#### 3. 内容单元识别规则

采用用户定制的逻辑组合，**配有手动修正能力**：

**核心规则（2026-07-13 最终修正）：**

1. **所有压缩包文件（.7z/.zip/.rar 等）** → 自动标记为内容单元
2. **文件夹** → 不自动标记，用户可手动标记
3. **手动标记文件夹时** → 其内部的所有内容单元标记自动取消

扫描仅产生**建议**，最终是否作为内容单元由用户决定。用户可通过右键"标记为内容单元"/"取消标记"随时修正。

示例：
```
Mods/Armor/Heavy/BDOR/  → 内部有 .7z → ✅ 内容单元候选（规则 1）
Mods/Armor/SomeMod.7z  → 单文件压缩包 → ✅ 内容单元候选（规则 2）
Mods/Animation/         → 内部无压缩包 → ❌ 分类目录
Mods/Armor/Test/        → 只有解压文件 → ❌ 不自动识别（可手动标记）
```

**显示原则**：中间列表显示目录下**所有**文件和文件夹，内容单元只是增加标记和元数据，不影响可见性。

**第一阶段限定：** 仅 Mod 类型的内容单元进入数据库。其他类型（教程、截图、图片等）暂不作为内容单元，不占用数据库元数据，直接通过文件系统浏览即可。理由：
- 教程：可用 Git 管理，元数据无必要
- 截图/图片：可直接看图，按文件名识别，不需要元数据膨胀数据库

#### 4. 扫描范围限定

- 受管理根目录（`managed_root`）仍保留，用于限定扫描范围
- 扫描时跳过已被识别为内容单元的目录的内部（避免重复统计文件）

# 第二步：新架构设计

根据六轮讨论的结论，梳理各层结构。

## Domain 层

### 实体定义

```text
ContentUnit（内容单元）
  - id: str
  - path: str              # 真实路径（可为中文）
  - title: str | None      # 别名/中文显示名，为空则显示文件夹名
  - content_type: str      # "mod"（第一阶段仅此一种）
  - source_url: str | None # 来源链接（如 N 网）
  - rating: int | None     # 评分（1-5）
  - cover_path: str | None # 封面图片路径（相对内容单元路径）
  - status: str            # 整理状态：unorganized / organized
  - notes: str | None      # 备注
  - created_at: str
  - updated_at: str

TagCategory（标签分类）
  - id: str
  - name: str              # 如"服装护甲"
  - color_hue: int         # 色相值，同一分类下标签颜色相近

Tag（标签）
  - id: str
  - name: str              # 如"重甲"
  - category_id: str

ContentUnitTag（内容单元 ↔ 标签，多对多）
  - content_unit_id: str
  - tag_id: str

OperationHistory（操作历史，简化版）
  - id: str
  - operation_type: str    # move / delete / rename / new_folder
  - source_path: str
  - target_path: str | None
  - created_at: str
  - can_undo: bool

ManagedRoot（保留现有）
  - id, real_path, path_key, display_name, created_at, updated_at

FolderCache（简化版 folder_node，目录树性能缓存）
  - id, path, parent_id, last_scanned_mtime, created_at

ThumbnailCache（保留现有）
  - asset_id, source_size_bytes, source_modified_at, cache_filename, status, ...
```

### 已移除的实体

- ❌ ModItem → 由 ContentUnit 替代
- ❌ FileAsset → 不再以"文件"为粒度记录（文件归属由目录表达）
- ❌ FileRole → 不再需要角色系统
- ❌ OperationLog（旧版四步状态机）→ 由简化版 OperationHistory 替代
- ❌ ConflictPolicy / OperationStatus / OperationType（旧版枚举）

### 路径重命名策略

- 推荐在应用内执行重命名（右键 → 重命名），由应用同步更新 ContentUnit.path
- 在应用外重命名后，下次扫描会标记旧路径"丢失"，需手动重新关联
