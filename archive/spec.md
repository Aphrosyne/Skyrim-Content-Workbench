# Skyrim Mod Workbench — 产品规格说明

## 1. 项目定位

Skyrim Mod Workbench 是一个面向 Windows 的、本地优先（local-first）的 Skyrim Mod 整理工作台。

它不替代 Mod Organizer 2，也不替代 Everything 或 Windows 资源管理器。它解决的核心问题是：用户积累了大量尚未分类的 Mod 文件，且一个逻辑 Mod 往往由本体、汉化、预览图、说明等多个文件组成；用户需要通过预览、中文信息和可视化目录树，快速完成识别、分组、分类与归档。

项目未来计划开源，第一版不包含云同步、账号系统或用户数据上传。

## 2. 核心原则

* local-first：所有数据默认保存在本机；不上传用户 Mod 文件、路径、图片或元数据。
* 文件系统是真实来源：Mod 文件始终是普通文件，能够脱离本程序继续被资源管理器、MO2 和其他工具使用。
* 数据库是管理层：数据库仅保存索引、中文信息、逻辑分组、预览关联、分类映射和操作日志。
* 安全优先：所有涉及用户 Mod 文件的操作必须可预演、可确认、可记录、可恢复。
* 人工确认优先：AI 只能生成建议，不能直接执行文件移动或修改。
* 中文优先：真实路径、中文目录、中文显示名、中文搜索均应正常工作。
* 不依赖统一命名规则：不得假设 Nexus Mods、本地汉化包、社区分享文件或预览图存在可靠命名规律。

## 3. 第一版目标

第一版的目标是帮助用户清理既有的大量未分类 Mod：

1. 扫描用户指定的 Mod 根目录和待整理目录。
2. 将压缩包、文件夹、图片、文档等文件显示为待处理素材。
3. 由用户手动创建“Mod 条目”，并把多个文件关联为一个逻辑组。
4. 为 Mod 条目填写中文名称、一句话用途说明、来源和分类。
5. 为 Mod 条目设置本地预览图作为封面。
6. 在可视化目录树中选择目标分类目录。
7. 将 Mod 条目的所有成员文件安全地整体移动到真实目标目录。
8. 导出待处理条目或未归类素材清单为 JSON，交给外部网页 AI 分类。
9. 导入 AI 返回的建议 JSON，在本地审阅并确认后应用。
10. 按中文名、原文件名、用途、分类和标签搜索已管理内容。

## 4. 非目标

第一版不包含以下功能：

* 云同步、账号、多人协作或远程数据库。
* 替代 MO2 的安装、排序、启用、冲突管理或游戏启动功能。
* 替代 Everything 的全盘文件搜索。
* 自动可靠识别 Mod 本体、汉化、预览图之间的关系。
* 不经用户确认的自动移动、删除、覆盖或重命名。
* 从 Nexus Mods 自动批量抓取预览图。
* 解析压缩包内部图片。
* 从应用向 Windows 资源管理器拖出真实文件。
* 文件哈希去重、完整文件监听、MO2 深度联动。
* 本地 AI 推理或 AI 直接控制文件系统。

## 5. 用户工作流

### 5.1 清库存主流程

1. 用户选择一个或多个本地根目录。
2. 程序扫描并建立索引。
3. 用户在“待处理素材池”中查看压缩包、图片、文档和文件夹。
4. 用户选中一个或多个素材，创建一个 Mod 条目。
5. 用户为成员指定角色：本体、汉化、预览、说明、可选文件或未知。
6. 用户填写中文名称、用途说明、来源和分类。
7. 用户设置一张预览图为封面。
8. 用户在左侧真实目录树中选择目标目录。
9. 程序先展示移动预演结果。
10. 用户确认后，程序移动该 Mod 条目的全部成员。
11. 程序写入操作日志，并提供撤销入口。

### 5.3 受管理根目录配置与手动扫描（阶段 2 Task 1 已实现）

1. 用户在主窗口点击「添加目录」，通过 Qt 原生目录选择对话框选择一个本地目录。
2. 程序将目录信息持久化到 `managed_root` 表（`real_path` 原样存储，`path_key` 标准化用于唯一约束），不扫描、不移动、不复制、不修改该目录或其中任何用户文件。
3. 用户关闭并重新打开应用后，已保存的根目录仍可见。
4. 用户在列表中选中一个根目录，点击「扫描选中目录」。
5. 程序在 Qt 后台线程执行 `FileScanner.scan()`，UI 不冻结；扫描期间禁用「扫描选中目录」与「添加目录」按钮。
6. 扫描完成后，程序展示摘要：扫描目录数、扫描文件数、持久化目录数、持久化文件数、错误数；若存在错误，展示前若干条错误摘要（路径与原因）。
7. 扫描失败（如根目录 ID 不存在、DB 异常）时，展示用户可读的错误信息。
8. 扫描严格只读，不改变用户文件；不实现文件监听、自动刷新、增量扫描、取消扫描。

### 5.4 只读目录树浏览（阶段 2 Task 2 已实现）

1. 主窗口中栏显示受管理目录树，数据源为 SQLite 中已持久化的 `FolderNode`，不在 UI 线程临时递归真实文件系统。
2. 树顶层节点包含两类：
   - 已配置且已扫描的受管理根目录（`ManagedRoot` 通过 `path_key` 关联到 `FolderNode`，显示为受管理根目录）；
   - 已配置但尚未扫描的根目录，显示为"未扫描"虚拟节点。
3. 用户可展开、折叠任意目录，查看其子目录；空目录正常显示。
4. 用户可选中任意目录节点，右侧详情区显示该目录的基础元数据：目录名称、完整真实路径、是否为受管理根目录、类型（受管理根目录/未扫描/普通目录）、直接子目录数量。
5. 扫描成功后，目录树自动刷新并展示扫描结果；切换根目录或重新扫描后树同步刷新。
6. 中文目录名和中文路径正确显示。
7. 目录树严格只读：不实现拖拽移动、右键文件操作、重命名、删除、新建文件夹、文件监听或自动刷新。
8. 符号链接、权限异常或扫描失败不会导致整个树崩溃；加载失败的分支记录日志并降级为空子树。
9. 重叠根目录的展示策略：若子根的 `path_key` 已被父根扫描覆盖（`FolderNode` 已存在），子根显示为"未扫描"虚拟节点，避免重复展示同一目录。
10. 重启应用后，目录树从已持久化的扫描数据加载，无需重新扫描真实文件系统。

### 5.5 未归类素材池与人工 Mod 条目组装（阶段 2 Task 3 已实现）

1. 主窗口中栏显示"未归类素材池"，列出所有 `mod_item_id` 为 `NULL` 的 `FileAsset`；数据源为 SQLite，不在 UI 线程重新扫描文件系统。
2. 素材池显示文件名和类型提示（文件 📄 / 文件夹 📁）；tooltip 显示完整路径。
3. 素材池支持多选；用户可选中一个或多个素材。
4. 用户点击"新建 Mod 条目"按钮，输入名称后创建一个空的 `ModItem`；新条目自动选中并出现在 Mod 条目列表中。
5. 用户在素材池多选素材后点击"关联到选中条目"，选中的素材以 `UNKNOWN` 角色关联到当前 `ModItem`；关联成功后素材从素材池消失，出现在成员列表。
6. 成员列表（右栏）表格显示每个成员的：文件名、类型（文件/文件夹）、角色（下拉可编辑）、完整路径、移除按钮。
7. 用户可通过角色下拉框为成员指定角色：`main_mod`（本体）、`translation`（汉化）、`preview`（预览图）、`readme`（说明）、`optional_file`（可选文件）、`unknown`（未知）。角色限制由 `ModAssemblyService.ROLE_LIMITS` 强制（`MAIN_MOD≤1`、`README≤1`，其他不限）；UI 不复制规则，直接展示服务层返回的错误。
8. 用户可点击成员行的"移除"按钮解除关联；解除后素材回到素材池，**不删除、不移动、不修改真实文件**。
9. 右栏显示当前选中 `ModItem` 的元数据编辑表单：显示名称、说明、来源链接、标签；点击"保存元数据"后写入数据库。
10. 关联失败原因（重复成员、角色超限、实体不存在等）以对话框或状态栏提示展示给用户。
11. 文件夹型与文件型 `FileAsset` 均可被关联和指定角色。
12. 中文文件名、中文显示名、中文标签、中文路径全程正确显示。
13. 扫描完成后素材池自动刷新，新扫描的未关联素材进入素材池。
14. 重启应用后，素材池、ModItem 列表、成员关联与角色、元数据均从已持久化数据加载。
15. 素材池严格只读边界：不实现忽略/删除/移出素材池机制（Q11 未决）；不实现搜索、缩略图、拖拽移动、自动分组、AI 建议。

### 5.2 AI 建议流程

1. 用户选择待处理 Mod 条目或未归类素材。
2. 程序导出 JSON。
3. 用户将 JSON 交给外部网页 AI。
4. AI 返回符合项目 JSON Schema 的建议文件。
5. 用户导入建议文件。
6. 程序显示建议差异，包括中文名、用途、分类、标签和候选成员关系。
7. 用户逐项或批量勾选确认。
8. 文件移动仍必须经过预演和确认。

## 6. 数据对象

### 6.1 ModItem

逻辑上的一个 Mod 条目，可包含多个真实文件。

字段：

* id
* display_name：中文显示名，可为空
* description：一句话用途说明，可为空
* source_url：来源链接，可为空
* category_folder_id：目标分类目录，可为空
* tags：标签集合
* status：阶段 2 不引入独立 status 字段（Q1 已关闭）。列表展示与可操作性由是否有关联成员、是否设置分类目录、是否有封面等现有字段表达
* cover_asset_id：封面预览图，可为空
* created_at
* updated_at

### 6.2 FileAsset

一个真实文件或文件夹。

字段：

* id
* mod_item_id：可为空
* real_path
* filename
* extension
* asset_kind：file 或 folder
* role：main_mod、translation、preview、readme、optional_file、unknown
* size_bytes
* modified_at
* imported_at
* batch_id：待确认，第一版是否实现下载批次概念

### 6.3 FolderNode

一个受管理的真实目录节点。

字段：

* id
* real_path
* parent_id
* display_name：可为空；若为空则显示真实目录名
* is_managed_root
* created_at
* updated_at

### 6.4 OperationLog

所有会影响真实文件的操作记录。

字段：

* id
* operation_type
* status：planned、confirmed、completed、failed、undone
* affected_asset_ids
* source_paths
* target_paths
* conflict_policy
* created_at
* completed_at
* undo_payload
* error_message

### 6.5 ManagedRoot

用户配置的受管理根目录。schema v2 引入。

与 `FolderNode` 的区别：`ManagedRoot` 保存用户配置，独立于扫描结果；`FolderNode` 表示扫描得到的目录树节点，`is_managed_root` 标识扫描时的根。移除 `ManagedRoot` 配置不自动清理对应 `FolderNode` 扫描记录（清理策略待确认）。

字段：

* id
* real_path：原样存储
* path_key：标准化路径键，用于唯一约束与比较（A2 决策）
* display_name：可为空；若为空则显示真实目录名
* created_at
* updated_at

## 7. 文件操作安全要求

所有文件移动必须遵守以下规则：

1. 前端不得直接操作文件系统。
2. 所有移动必须由统一的文件操作服务执行。
3. 执行前必须生成预演结果。
4. 预演必须列出每个源路径、目标路径、是否跨盘、是否重名、是否缺失、是否不可写。
5. 用户必须显式确认预演结果。
6. 默认冲突策略为“询问”；不得默认覆盖。
7. 禁止把文件夹移动到其自身或子目录。
8. 同盘移动优先使用原子移动或重命名。
9. 跨盘移动必须明确提示其为复制后删除流程。
10. 每个成功移动必须记录源路径和目标路径。
11. 撤销前必须再次检查当前文件状态；不安全时不得强行回退。
12. 单个成员移动失败时，不得将整个 Mod 条目标记为完全成功。
13. 不得删除用户文件；第一版不提供删除功能。
14. 不得修改用户文件内容。
15. 所有预览缓存必须与用户 Mod 文件目录分离。
16. 部分成员失败时不自动回滚已成功成员（Q20 已关闭）；UI 必须展示成功与失败成员，用户可对已成功部分发起撤销预演。

## 8. UI 结构

主窗口采用三栏布局：

* 左栏：受管理根目录列表 + 添加目录 + 扫描选中目录按钮（阶段 2 Task 1 已实现）+ 扫描状态 + 受管理目录树（阶段 2 Task 2 已实现只读浏览）+ 选中目录详情区。目录树显示受管理根目录及其子目录，可展开、折叠；支持中文真实目录和中文显示名。目录树以数据库中的 `FolderNode` 为数据源，不是可写的即时文件系统树。选中目录树节点后显示目录名称、完整真实路径、是否为受管理根目录、类型、直接子目录数量。
* 中栏：未归类素材池 + ModItem 列表 + 新建/关联按钮。素材池列出 `mod_item_id` 为 `NULL` 的 `FileAsset`，每行显示文件名、类型（文件/文件夹）和完整路径，支持多选（阶段 2 Task 3 已实现）。ModItem 列表显示全部 ModItem，选中后右栏加载详情与成员。
* 右栏：ModItem 详情编辑 + 成员列表。选中 ModItem 后显示元数据编辑表单（显示名称/说明/来源链接/标签 + 保存按钮）与成员表格（文件名/类型/角色下拉/路径/移除按钮）（阶段 2 Task 3 已实现）。

素材池行为（阶段 2 Task 3）：

* 素材池仅显示 `mod_item_id` 为 `NULL` 的 `FileAsset`，数据源为 SQLite，不在 UI 线程重新扫描文件系统。
* 素材池每行显示：图标（📁/📄）+ 文件名 +（类型）+ 完整路径；tooltip 同步显示类型与路径。
* 支持多选（Ctrl/Shift）；「新建 Mod 条目」和「关联到选中条目」按钮在无选中素材时禁用。
* 用户在素材池多选素材后点击「新建 Mod 条目」：输入名称后创建 ModItem，选中的素材自动以 `UNKNOWN` 角色关联到该条目。
* 用户也可先选中一个已有 ModItem，再在素材池多选素材，点击「关联到选中条目」将素材关联到该 ModItem。
* 关联成功后素材从素材池消失，出现在右栏成员表格；解除关联后素材回到素材池。
* 不实现忽略/删除/移出素材池机制（Q11 未决）。

目录树只读边界（阶段 2 Task 2）：

* 目录树仅用于浏览和选择；不实现拖拽移动、右键文件操作、重命名、删除、新建文件夹、文件监听或自动刷新。
* 目录树数据源严格为 SQLite `FolderNode`；不在 UI 线程临时递归真实文件系统。
* 已配置但尚未扫描的根目录显示为"未扫描"虚拟节点，不假装为空目录。
* 加载失败的分支记录日志并降级为空子树，不让整个树崩溃。

阶段 2 移动入口为按钮式「移动到选中目录」（Q3 阶段 2 部分已关闭）；不实现卡片拖入目录树、树内拖拽或从资源管理器拖入。拖拽为后续阶段待确认项。

阶段 2 不实现搜索栏、卡片图库、缩略图生成、AI JSON、联网预览图导入。

## 9. 搜索

第一版搜索范围：

* Mod 条目中文名
* 原始文件名
* 用途说明
* 来源链接
* 标签
* 真实路径
* 分类目录显示路径

搜索仅针对受管理目录和本地数据库，不承担全盘搜索职责。

## 10. 预览图

第一版预览图要求：

* 支持将本地图片关联到 Mod 条目。
* 支持从素材池选择图片作为封面。
* 支持常见图片格式，至少包括 JPG、PNG、WEBP。
* 预览图可以是 Mod 条目的真实成员，也可以是外部关联图。
* 应用生成的缩略图缓存必须保存到应用数据目录。
* 不得修改原始图片文件。

阶段 2 Task 4 已实现范围：

* 将 PREVIEW 角色成员设为 ModItem.cover_asset_id（非 PREVIEW 成员被拒绝）。
* 为已关联的 preview 本地图片生成缩略图，缓存写入 `%LOCALAPPDATA%\SkyrimModWorkbench\thumbnails\`。
* 缩略图缓存以 `asset_id` 命名（`{asset_id}.png`），元数据存入 `thumbnail_cache` 表（schema v3）。
* 缓存有效性基于 `asset_id + source_size_bytes + source_modified_at`；不一致时过期重建。
* ModItem 列表显示封面缩略图图标与成员数；详情区显示封面预览与 preview 成员。
* 图片不可读、格式不支持、源文件不存在时显示安全占位状态，不崩溃。
* 缩略图生成在后台线程执行，不卡死 UI；worker 在自身线程内创建独立 SQLite 连接。
* 只读访问原图；不修改、不转换、不压缩、不覆盖用户原图。
* 支持中文路径与常见图片格式（JPG/JPEG/PNG/WEBP/GIF/BMP/TIF/TIFF/ICO）。

待确认：第一版是否允许"外部关联图"不复制到应用缓存，只保存原路径。
未实现（归阶段 5 预览增强）：预览图墙、手动从本地添加多张预览图、Nexus URL 导入预览图、OCR、图像识别。

## 11. AI JSON 交换

AI 仅处理摘要数据，不获得本地文件操作权限。

导出内容至少包括：

* Mod 条目 ID 或素材 ID
* 文件名
* 扩展名
* 文件角色
* 当前分类
* 中文名和用途（如已有）
* 是否有预览
* 来源链接
* 可选的分类目录列表

导入建议至少支持：

* 建议中文名
* 建议用途说明
* 建议分类目录 ID
* 建议标签
* 候选成员关系
* 置信度
* 原因说明

待确认：是否在第一版导出预览图缩略图联系表；这会提高 AI 判断能力，但增加导出复杂度。
